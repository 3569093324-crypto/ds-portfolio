## 案例 1: 用户成长体系指标设计

### 北极星指标（1个）

**"DAU中完成至少1个成长任务的用户占比"**

为什么不选"等级提升用户数"？→ 等级提升是长期结果，北极星需要能高频观测。
为什么不选"积分获取总量"？→ 只关注行为不关注生态，可能诱导刷分。

### 过程指标（5个）

按用户旅程设计 — AARRR 框架：

| 阶段 | 指标 | 为什么重要 |
|------|------|-----------|
| **Acquisition（触达）** | 成长体系入口曝光率、点击率 | 用户是否知道这个功能存在 |
| **Activation（激活）** | 完成首个任务的新用户比例 | "首次体验"决定是否留下 |
| **Retention（留存）** | 7日/30日任务完成留存率 | 是否形成习惯 |
| **Revenue（变现）** | 等级提升后的ARPU变化 | 成长体系是否带来了商业价值 |
| **Referral（传播）** | 分享成就/等级的用户比例 | 社交裂变潜力 |

### 护栏指标（3个）

| 指标 | 阈值 | 为什么 |
|------|------|--------|
| 核心功能使用时长 | 不降超过5% | 防止用户为了积分而忽略了产品核心价值 |
| 客服投诉率 | 不升超过10% | 等级特权可能导致非付费用户不满 |
| 刷分检测率 | <0.1% | 防止黑产利用体系漏洞 |

### 面试话术

> "我设计指标的原则是：北极星指标要能高频观测且直接反映成长体系的健康度；
> 过程指标沿用户旅程展开，确保不遗漏转化漏斗的任一步；
> 护栏指标防止'好心办坏事'——成长体系提升活跃度但可能伤害核心体验。"

## 案例 2: Search vs Recommendation 效果评估的区别

### 用户意图的本质差异

| | Search（搜索） | Recommendation（推荐） |
|---|---|---|
| **用户意图** | **明确**（我知道我要什么） | **模糊**（我不知道我要什么/随便看看） |
| **行为模式** | 输入query→浏览结果→点击/购买 | 被动接收内容→被吸引点击 |
| **成功定义** | 快速找到想要的 | 发现感兴趣但没想到的内容 |

### 核心指标对比

| 指标 | Search | Recommendation |
|------|--------|----------------|
| **核心转化** | Query→Click Rate (CTR at position) | Exposure→Click Rate |
| **效率** | Time-to-Find, 搜索修改率 | 多样性（不同类目曝光率） |
| **满意度** | 搜索结果相关性评分 | 惊喜度（Serendipity） |
| **覆盖** | 零结果率 | 冷启动覆盖率 |
| **长期** | 重复搜索率（第一次没找到） | 探索-利用平衡 |

### 评估方法的差异

| 维度 | Search | Recommendation |
|------|--------|----------------|
| **离线评估** | NDCG, MAP, MRR（有ground truth） | 无明确label → 依赖用户行为模拟 |
| **A/B测试** | 单一query可以隔离测试 | 用户级别的随机化 |
| **评估难度** | 较低（有正确答案） | 较高（没有唯一正确答案） |

### 面试话术

> "Search和Recommendation的最大区别是用户意图——搜索是'你知道你要什么'，
> 推荐是'你也不知道你要什么但看到了觉得不错'。所以搜索的指标偏向效率（快点找到），
> 推荐的指标偏向发现性（多看看不同的）。离线评估时搜索有明确的ground truth（MAP/NDCG），
> 推荐很难在离线状态判断'这个推荐好不好'。"

## 案例 3: 时长↑5%但满意度↓ — Trade-off分析

### Step 1: 确认因果链路

首先要问：**时长增加和满意度下降是因果关系还是相关性？**

可能场景：
- A) 产品改版让用户更难找到想要的内容→花更多时间找→满意度降（因果）
- B) 新增了很多内容→用户愿意花时间但内容质量参差不齐→满意度降
- C) 两者无因果关系，只是同期发生了其他变化

### Step 2: 实验设计验证

```
实验假设: "功能X导致使用时长增加但满意度下降"

对照组: 不使用功能X的用户
实验组: 使用功能X的用户

要测量的:
1. 时长变化（主指标）
2. 满意度变化（主指标）
3. 关键行为序列: 用户在增加的时长里做了什么？
   → 更多浏览新内容？
   → 重复刷新无结果页面？
   → 在填写复杂表单？
```

### Step 3: 三分法分析

| 用户群 | 时长 | 满意度 | 占比 | 解读 |
|--------|------|--------|------|------|
| A) 探索型 | ↑ | ↑ | 30% | 好: 发现更多有价值内容 |
| B) 困惑型 | ↑ | ↓ | 45% | 坏: 找不到想要的东西 |
| C) 不受影响 | - | - | 25% | - |

→ 发现B群占比最高 → 时长增加是因为效率降低

### Step 4: 深层诊断

- 页面加载时间是否变慢？
- 搜索准确率是否下降？
- 新用户比例是否增加（通常新用户效率更低）？
- 用户最常在哪个环节花时间？

### Step 5: 决策建议

| 情况 | 建议 |
|------|------|
| 时长增加来自探索（A群为主） | 上线，增加内容多样性 |
| 时长增加来自困惑（B群为主） | 不优化效率就上线→饮鸩止渴 |
| 短期时长↑但长期留存↓ | 绝对不能上线 |

### 面试话术

> "时长不是越长越好。如果多出来的5%是因为用户找不到东西在反复搜索，
> 这种'时长增长'反而是产品体验恶化的信号。我需要把用户分层分析：
> 探索型增长是好事，困惑型增长是坏事。决策取决于哪种用户占主导。"

## 指标设计的通用原则

```
1. 指标金字塔
   北极星指标（1个）
   ├── 一级指标（3-5个）：核心业务环节
   ├── 二级指标（10-20个）：可操作的杠杆
   └── 护栏指标（3-5个）：不可逾越的底线

2. HEART框架 (Google)
   H - Happiness（满意度/NPS）
   E - Engagement（活跃度/时长）
   A - Adoption（新用户采纳率）
   R - Retention（留存率）
   T - Task Success（任务完成率/效率）

3. 好指标的特征
   ✅ 可衡量、可比较、可行动
   ✅ 有明确的责任归属
   ✅ 不与其他指标严重冲突
   ❌ 虚荣指标（PageView——内容丰富不代表价值高）
```

---

*指标设计核心：不是"越多越好"，而是"在正确的时间看正确的指标"。*
