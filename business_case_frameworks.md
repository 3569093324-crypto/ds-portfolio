## 案例 1: DAU 突然下降 20%

### 框架: 维度拆解 → 时间定位 → 内外部因素 → 假设验证

#### Step 1: 问题界定和确认
- "20%" 是环比（昨天vs前天）还是同比（vs上周同一天）？
- 什么时间段的数据？确认数据口径是否一致
- 是否排除数据管道问题（埋点丢失、ETL延迟）？

#### Step 2: 维度拆解（找到下降在哪里）

| 维度 | 拆分方式 | 检查什么 |
|------|---------|---------|
| 平台 | iOS vs Android vs Web | 是否某个平台崩溃/发版bug |
| 地区 | 按省市拆分 | 是否局部网络故障/天气影响 |
| 渠道 | 自然流量/付费/社媒/推送 | 是否某个渠道失效 |
| 用户群 | 新用户 vs 老用户 | 新用户获取骤降？老用户流失？ |
| 版本 | App版本号 | 新版本有bug导致闪退 |

关键问题: "20%的下降是均匀分布还是集中在某个维度？"

#### Step 3: 时间定位
- 精确到小时：是某个小时的骤降还是全天均匀下降？
- 趋势图：过去7天的DAU曲线 → 是新现象还是趋势延续？

#### Step 4: 内外部排查
**内部**:
- 技术: 服务器宕机？CDN故障？数据库慢查询？发版导致闪退？
- 产品: 昨天上线了什么功能？改了登录流程？
- 运营: 推送服务故障？活动结束的预期回落？

**外部**:
- 竞品: 竞品昨天搞大促/发布新品？
- 环境: 节假日？天气？新闻事件？

#### Step 5: 假设-验证逻辑链
```
观察 → 假设 → 数据验证 → 结论 → 行动

例: "如果因为iOS新版本闪退，我们应该看到：
     1) iOS用户下降幅度远大于Android
     2) 下降集中在昨天发版之后
     3) 新版crash rate上升
     4) 低版本用户DAU不受影响
     如果这些都为真 → iOS版本bug → 紧急回滚"
```

## 案例 2: 商品详情页转化率连续 3 周下降

### 框架: 流量 → 页面 → 用户 → 竞品 → 外部

#### Step 1: 流量质量分析
- 各渠道转化率变化？付费渠道拉新质量可能下降
- 关键词/搜索词变化？进来的用户意图是否变了
- CPC/CPA是否有异常？

#### Step 2: 页面体检
- 加载时间是否变慢？（每+1s转化率-7%）
- 最近3周UI/UX改动？（即使是小改动）
- 价格/库存/评论有变化？
- A/B测试：是否当前有其他测试影响？

#### Step 3: 用户群分析
```
用户 = 新用户 + 老用户

如果新用户转化率↓而老用户不变 → 流量质量问题
如果老用户也↓ → 页面/产品本身有问题
如果都在↓ → 可能是宏观因素
```

- 新老用户转化率分别看
- 不同设备/浏览器分别看
- 用户等级（VIP/普通）分别看

#### Step 4: 竞品调研
- 竞品降价/推出同类产品？
- 竞品页面体验显著提升？

#### Step 5: 外部/季节性
- 是否是行业淡季？
- 天气/节假日影响？
- 宏观经济变化（消费意愿下降）？

#### 输出: 排查优先级矩阵
```
高影响+高概率 → 首先排查 (如页面加载时间)
高影响+低概率 → 做好准备
低影响+高概率 → 次优排查
低影响+低概率 → 暂时搁置
```

## 案例 3: A/B测试不显著，但PM想上线

### 框架: 统计视角 → 业务视角 → 风险视角 → 决策建议

#### Step 1: 统计视角 — 为什么可能不显著？

**检查清单**:
1. 样本量够吗？→ Power Analysis 反算
2. 实际效应量多大？→ 即使不显著，方向对+效应量不小
3. 置信区间有多宽？→ CI包含0但上界很高→样本不够
4. 方差是否太大？→ 指标本身波动大
5. 实验时间是否够长？→ 新奇效应、周末效应

> "p=0.07不显著，但效应量为+2.3%且CI上界=+5.1% → 这可能是power不足，而非无效果。"

#### Step 2: 业务视角 — 不显著≠没价值

| 情况 | 建议 |
|------|------|
| 效应量方向为正 + CI合理窄 | 扩大样本再测 |
| 效应量几乎为零 | 新方案确实无效，放弃 |
| 效应量为负 | 不能上线，可能有负面影响 |
| 次要指标显著变好 | 考虑调整主要指标 |

#### Step 3: 风险视角 — 上了最坏会怎样？

- 如果不显著的新方案上线，最差是什么？
  - 最差：降低转化率 → 直接收入损失
  - 最好：其实有效→获得收益
- **反转测试 (Reverse Test)**: 如果新方案有害的概率是多少？
  - 如果p=0.07（双尾），单尾p=0.035 → 有96.5%概率至少不差

#### Step 4: 决策建议

```
✅ 可以接受上线的条件:
  1. 效应量方向正确且幅度有业务意义
  2. 上线代价低（容易回滚）
  3. 下线方案有明确的监控指标和回滚阈值
  4. 次要指标/护栏指标无恶化

❌ 不应上线的条件:
  1. 效应量方向为负
  2. 上线后不可逆
  3. 涉及核心转化路径（风险太大）
  4. p > 0.2且效应量极小
```

**面试话术**: "我会建议做一次功效分析，确认样本量是否足够检测到业务上重要的最小效应。如果power足够但效应量很小→放弃。如果power不足→扩大样本再测2周。决不会因为'PM想做'就跳过统计流程。"

## 面试中的通用分析框架

### 业务指标异常的通用框架 (3A)

```
1. AFFIRM (确认问题)
   - 确认指标口径、时间窗口
   - 排除数据问题（缺失、重复计数）

2. ANALYZE (分析归因)
   - 内外部因素扫描
   - 维度拆解定位（平台/地区/渠道/人群）
   - 同期群分析

3. ACT (提出方案)
   - 短期止血
   - 长期改进
   - 监控预警
```

### PSI 面试回答公式

- **P (Problem)**: 复述问题，确认理解正确
- **S (Structure)**: 给出分析框架（不是直接给答案）
- **I (Insight)**: 基于框架给出可能的原因和建议

---

*面试关键：先给框架再给细节，展示思考过程而非猜测答案。*
